Plugin Axis Template

The Plugin Axis template allows drive vendors to create SoftMotion axis support for their EtherCAT drive. This template assists the plugin writer in developing a set of VIs that enable communication between SoftMotion and an EtherCAT drive.

System Requirements

  • LabVIEW Base, Full, or Professional Development System
  • LabVIEW Real-Time Module
  • LabVIEW SoftMotion Module
  • NI-Industrial Communications for EtherCAT

Use Cases

The plugin axis template is designed to be used by drive vendors to create SoftMotion support for their hardware. Once a drive vendor has created support for their hardware, LabVIEW and SoftMotion customers will use the plugin axis type to interact with their EtherCAT drive.

Overview

To create a plugin axis, begin with the provided template project. The template provides you with the following core components:

  • Getting Started VI— An example VI that demonstrates the use of a plugin axis. This can also be used to test your plugin.
  • Template Plugin Class — The base class from which all plugin axes are derived. Your plugin requires a derived class that implements the following two methods:
    1. Open.vi— This VI is used to register your plugin with SoftMotion. The Open.vi must be placed on the user's block diagram for every plugin axis that will be used. This VI instantiates your plugin axis class. NI recommends including this VI in a palette specific to your plugin.
    2. GetBindings.vi— This VI is used by SoftMotion to add your axis type to the SoftMotion Axis Manager dialog. By providing a VendorID and a ProductID, SoftMotion can search for compatible hardware in the project and add it as an axis binding.
      Note  The filename for your GetBindings.vi must include the text GetBindings in order to be located by SoftMotion.

Your plugin must be installed into a uniquely named folder within <LabVIEW>\vi.lib\Motion\plugins\. Use a descriptive folder name to identify the hardware you support.

NI strongly recommends creating an example VI demonstrating the use of your plugin, similar to the provided Getting Started VI.

A plugin that adheres to the DS 402 specification implements the following features:

  • Enabling/Disabling drive
  • Moves (Axis and Coordinate)
  • Cyclic Sync Position
  • Velocity Profile
  • Torque Profile
  • Clear fault
  • Read fault
  • Position capture (Touch Probe)
  • Digital IO

Base Class

The base class (nismPluginAxisBaseClass.lvclass) implements a set of methods that can be overridden by your plugin. It is not required to override any of the base class methods. A working plugin only requires Open.vi and GetBindings.vi. The following list describes the base class methods with examples of when you might need to override them.

Note   Any base class methods that you override must have a name identical to the parent method or your overrides will not be called (dynamic dispatch).

OnTransitionToConfigMode.vi

This VI will perform an action or actions in Active mode before the scan engine transitions to Configuration mode. These actions will only execute if the user sets the scan engine mode using nismPluginSetPluginMode.vi. The base implementation will stop any move in progress and disable the drive.

OnTransitionToActiveMode.vi

This VI will perform an action or actions in Configuration mode before the scan engine transitions to Active mode. These actions will only execute if the user sets the scan engine mode using nismPluginSetPluginMode.vi. Use this to send any SDOs that must be set when the scan engine is in Configuration mode. The base implementation is empty.

Initialize.vi

This VI does any initialization that is necessary for the plugin. It runs when the scan engine is in active mode. The base implementation does not do anything.

Execute.vi

This VI manipulates PDO data. It gets data from SoftMotion (such as setpoints or controlword) and manipulates it before sending the data to the EtherCAT slaves. It also gets data from the EtherCAT slave (such as statusword or feedback) and manipulates it before sending it back to SoftMotion.

The base class implements a specific set of PDOs in a specific order, and packs the data to work with the EtherCAT driver. No scaling is implemented. To scale data or have a different PDOs layout, you must override this method.

SoftMotion assumes that any rollover will happen at the datatype limit. For example, the expected behavior of the Position Actual Value PDO is to wrap around when the value exceeds 231 and begin incrementing from -231. If your EtherCAT slave does not work like this, you must modify your Execute.vi to enforce this behavior.

The following PDOs are implemented by the base class:

Octet Write PDO
0-1 Controlword 0x6040
2-5 Target Position 0x607A
6-9 Target Velocity 0x60FF
10-11 Target Torque 0x6071
12-15 Digital Output 0x60FE
16-17 Touch Probe Function 0x60B8

Octet Read PDO
0-1 Statusword 0x6041
2-5 Position Actual Value 0x6064
6-9 Velocity Actual Value 0x606C
10-11 Torque Actual Value 0x6077
12-15 Digital Input 0x60FD
16-17 Touch Probe Status 0x60FD

SetProperty.vi

This VI is used to send SDOs to the EtherCAT slave. Since it runs in a time critical loop, it must use the EtherCAT driver’s asynchronous SDO API. This VI should only be used to service SDOs initiated by SoftMotion. The plugin framework does not handle non-SoftMotion initiated SDOs. These must be handled elsewhere by the plugin writer. This VI works as follows:

  1. SoftMotion toggles the Set input to TRUE for one cycle when sending a new SDO.
  2. The VI sends the SDO, then checks its status. It will check the status every time the VI is called.
  3. The Done boolean is set to TRUE to notify SoftMotion that the transfer was completed.

The base class handles specific SDOs. You must override SetProperty.vi if your hardware does not support these SDOs or handles them differently. The supported SDOs are as follows:

Note   This list may be incomplete, and may change with future releases of SoftMotion.
SDO Index Subindex
Modes of Operation* 0x6060 0
Interpolation Time Period 0x60C2 1, 2
Digital Outputs† 0x60FE 2
Homing Method 0x6098 0
Homing Speeds 0x6099 1, 2
Homing Acceleration 0x609A 0
* SoftMotion can be put into Profile Velocity Mode (3), Profile Torque Mode (4), Homing mode (6), and Cyclic Position mode (8).
† Subindex 1 is handled as a PDO.

GetProperty.vi

This VI is used to read SDOs from the EtherCAT slave. Since it runs in a time critical loop, it must use the EtherCAT driver’s asynchronous SDO API. This VI should only be used to service SDOs initiated by SoftMotion. The plugin framework does not handle non-SoftMotion initiated SDOs. These must be handled elsewhere by the plugin writer. This VI works as follows:

  1. SoftMotion toggles the Get input to TRUE for one cycle when requesting a new SDO.
  2. The VI requests the SDO, then checks its status. It will check the status every time the VI is called.
  3. The Done boolean is set to TRUE to notify SoftMotion that the transfer was completed.

The base class handles specific SDOs. You must override GetProperty.vi if your hardware does not support these SDOs or handles them differently. The supported SDOs are as follows:

Note   This list may be incomplete, and may change with future releases of SoftMotion.
SDO Index Subindex
Error Code 0x1003 0-10
Touch Probe Pos1 Pos Value 0x60BA 0
Touch Probe Pos1 Neg Value 0x60BB 0

ErrorHandler.vi

This VI is used to handle any errors that may arise from your plugin. It is called once per scan period. The base class does not do any error handling, and passes through the error.

Cleanup.vi

This VI is used to perform any cleanup that may be required. It runs when the scan engine is in Active mode. The base class does not implement any cleanup routine.

Accessor Methods

There are several accessor methods to read and write the base class properties.

Using Plugin Axes

The Getting Started VI included in the template project demonstrates the use of a plugin axis. There are some significant differences between using plugin axes and standard SoftMotion axes. Plugin axes may not be run immediately on booting an RT target; the plugin framework must be launched first.

Open.vi

The first step when running a plugin axis is to pass the axis reference to the plugin Open.vi. Every plugin will have an Open.vi that instantiates an instance of the plugin class and sets the axis resource on that class. The axis resource is required to get the slave ID of the EtherCAT slave associated with that axis.

The output of Open.vi is a reference to the base class. To start a plugin axis, bundle all base class references into an array and call nismPluginSetPluginMode.vi. This VI can only be called on the diagram once, so pass all plugin class references at one time. When this VI is called with the Mode input set to "Active", the plugin framework begins running. To stop execution, call this VI with the Mode input set to "Configuration".

Stages of Framework Execution

  • Query the EtherCAT bus for slaves— The plugin framework will only search for plugin axis slaves.
  • Enable Raw Data Mode— Using I/O variables for EtherCAT communication can be slow. The EtherCAT raw data API has better performance.
  • Get EtherCAT Bus Data— The EtherCAT raw data API requires certain information about the bus, such as data offsets and data size.
  • Launch the Process VI— The process VI runs the timed loop responsible for EtherCAT communication such as sending PDOs and SDOs. It also runs the plugin VIs.

Modifying this Template

Because they run in a timed loop, refer to the Real-Time Module best practices in the LabVIEW Help when modifying the following VIs:

  • GetProperty.vi
  • SetProperty.vi
  • Execute.vi
  • ErrorHandler.vi

The VIs located at <LabVIEW>\2014\vi.lib\Motion\plugins\PluginFramework must remain unmodified for the plugin framework to function correctly.
